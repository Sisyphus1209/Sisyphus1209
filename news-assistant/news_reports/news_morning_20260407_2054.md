### 1. 赫尔墨斯公司融资3.5亿美元，用于研发自主超音速战斗机。
*TechCrunch* | 12:30

美国国防初创公司赫米乌斯（Hermeus）已融资3.5亿美元，用于继续开发其所谓的"最快无人机"。本轮融资使其估值达到10亿美元。这家总部位于洛杉矶的初创公司周二表示，已通过股权融资筹集了2亿美元，由科斯拉创投领投。现有投资者迦南合伙公司、创始人基金、In-Q-Tel和RTX创投也参与了本轮融资。新的外部资金来自媒体集团考克斯企业的风险基金、公开交易的封闭式管理投资公司Destiny Tech100等。

剩余的1.5亿美元以债务形式提供。赫米乌斯联合创始人兼首席执行官AJ·皮普利卡告诉TechCrunch，这将有助于这家初创公司及其不断增长的资本结构保持一定的控制权。他在接受采访时表示："我们制造了大量硬件，正在扩大制造能力，如果我们能以非稀释性方式为大部分支出融资，这绝对是正确的做法。"

赫米乌斯融资之际，风险投资和公司投资者正将资金大量投入国防初创公司。根据PitchBook的数据，去年全球国防科技领域的风险投资超过90亿美元，涉及265轮融资，其中公司投资者通过28轮融资贡献了20亿美元。但对赫米乌斯来说，这不仅仅是时机好。

皮普利卡将至少部分融资成功归功于赫米乌斯几年前在技术方面做出的改变。他说，这家初创公司花费了时间和资金开发自己的发动机，部分是出于必要。在赫米乌斯向RTX公司（前身为雷神公司）的风险投资部门RTX创投示好之后，新的机会出现了。

皮普利卡和他的团队决定与RTX的子公司普惠公司合作，改造这家航空航天公司的F100发动机，为赫米乌斯的高超音速飞机提供动力。这使赫米乌斯走上了一条更快的轨道，拥有经过验证且功能强大的发动机，使其更容易进行测试和迭代，同时在此过程中与美国政府签订新合同。

赫米乌斯总裁扎克·肖尔表示，现在赫米乌斯能够实现多样化，而不是瞄准制造5马赫飞机的单一宏大目标。他说："这加速了我们实现5马赫的目标，同时也加强了业务的经济性，同时满足了国防部近期的需求。我认为，通过这种方式，许多同心圆同时重叠，加强了业务，加强了客户，也加强了技术的成熟。"

上个月，赫米乌斯试飞了其技术的演示版本，其尺寸与F-16战斗机相当。这家初创公司表示，其目标是让该飞机的下一次迭代实现超音速。皮普利卡说，第三架飞机也在研制中。

皮普利卡表示，这种快速原型设计方法在航空领域很难实现。他指出，SpaceX是行业标准，愿意建造、测试、失败、学习并重复，直到造出正确的飞行器。这就是为什么赫米乌斯面临的最艰巨挑战是培养或开发人才。

他说："世界上没有哪家公司每年都在制造新的全尺寸飞机，无论是全新设计还是其他方式。人们过去常常这样做，但他们都去世了，这意味着你必须以某种方式培养这些人。"新一轮融资也将帮助赫米乌斯继续扩大员工规模，目前员工人数已接近300人。

赫米乌斯现在已经成功进行了两次试飞（去年试飞的演示机尺寸小了三分之二）。但皮普利卡强调，赫米乌斯需要为某种失败做好准备——他再次将其视为快速原型设计过程的一部分。他说："挑战在于，如何选择正确类型的风险块，并随着时间的推移将资本应用于这些风险块。是的，我们可能会坠毁一架飞机，我预计这会在我们开发计划的某个阶段发生。我们已经做好了非常安全的准备。但这也是为什么制造更多飞机非常重要。如果你不制造很多飞机，你会花费更长的时间，因为你会小心翼翼地对待它们。你知道，我们想知道为什么开发一架新飞机需要20年、25年？"

### 2. Spotify的提示播放列表功能现已支持播客内容。
*TechCrunch* | 12:00

Spotify今日宣布，其AI歌单生成功能"提示歌单"将扩展至播客领域。该公司去年底在新西兰率先测试该功能，初期专注于音乐歌单创建。今年已陆续在美国、加拿大、英国、爱尔兰、澳大利亚和瑞典等市场推出。

目前，这些市场的付费用户可通过测试版功能创建AI辅助的英文播客歌单。用户点击"创建"并选择"提示歌单"选项后，可输入如"创建我可能感兴趣的犯罪纪实播客歌单，添加我可能错过的高评分系列，尤其要包含情节曲折的作品"等提示指令。

用户还可设置歌单更新频率，选项包括每日更新、每周更新（可指定具体日期）以及不更新。该公司表示，当特定单集被添加至歌单时，会附简短说明解释推荐理由。

Spotify全球播客内容总监莉兹·黑尔在声明中表示："播客爱好者始终在寻找优质内容。提示歌单让内容发现变得轻松个性化。对创作者而言，这开启了全新的曝光机遇，既能将经典作品又能将新内容推送给主动表达收听意愿的听众。"

数据显示，每周有超过3400万播客节目在Spotify上首次被用户发现。该公司正通过这项AI新功能加强播客内容推荐。用户可能在首页看到由Spotify文化团队创建的提示歌单，这些歌单可通过修改提示词轻松自定义。

这家音频流媒体平台持续优化播客生态。今年早些时候，其降低了视频播客的盈利门槛。新规要求创作者需发布至少3个播客节目，过去30天收听时长达2000小时，且最近30天拥有至少1000名活跃听众。今年2月的2025年第四季度财报电话会议上，联席CEO亚历克斯·诺斯特罗姆透露，平台视频播客数量已突破53万。

### 3. 可折叠iPhone模型曝光，传闻发布时间或推迟
*The Verge* | 11:27

在爆料者兼记者桑尼·迪克森分享了据称是可折叠iPhone模型机的照片后，我们或许首次得以窥见这款异常宽大的iPhone Fold的形态。这些照片中还包括了iPhone 18 Pro和18 Pro Max的类似模型。这一明显的外观设计泄露之际，有报道称这款可折叠设备的早期工程测试遇到了生产问题，可能导致其发货时间比今年其他iPhone晚数月。

**宽屏可折叠iPhone模型现身，延迟发布传闻四起**

iPhone Fold与iPhone 18 Pro模型机同框亮相，但可能晚数月发布。

迪克森今早在Bluesky上分享了这款可折叠设备的图片。他以准确获取苹果和三星手机模型机（供手机壳制造商测试和设计产品使用）而闻名，有着良好的记录。图片展示了手机的正面和背面，尽管模型内部并未透露太多可能的设计细节，甚至没有展示前置摄像头或Face ID传感器的可能布局。外部则提供了更多信息，显示两个后置摄像头并排位于一个凸起的药丸形模组中，该模组横跨了这款异常宽大机身约三分之二的宽度。

迪克森没有提供模型机的确切尺寸，但它看起来比我们迄今为止见过的任何书本式可折叠设备都更矮更宽。其风格最接近谷歌的第一代Pixel Fold，但似乎比那款手机更宽。这一选择可能看起来有些奇怪，但据传三星也在研发一款"宽屏折叠"变体，计划于今年晚些时候发布，这表明这种外形设计可能具有市场潜力。

问题可能在于哪家公司会率先推出。外界普遍预计苹果将在9月的常规iPhone发布会上首次推出可折叠iPhone，与18 Pro和18 Pro Max一同亮相，而基础款iPhone 18则将于2027年发布，但这一时间表可能存疑。《日经亚洲》报道称，早期生产遇到了"工程挑战"，解决时间比预期更长，"可能导致首批发货延迟数月"。

一位消息人士告诉《日经》："苹果和供应链正在紧张的时间表下工作，目前的解决方案不足以完全解决工程挑战。需要更多时间。"这一报道的延迟与彭博社马克·古尔曼最近的预测相呼应，即这款可折叠设备将在两款Pro版iPhone之后的一段时间内发货，即使它与它们一同发布。

### 3. 比尔·菲利普斯如何用流动水模拟经济运行

比尔·菲利普斯如何用流动的水模拟经济
亲爱的读者，我们想与您分享新书《金钱星球：塑造你生活的经济力量指南》中的一段精彩节选。该书由亚历克斯·迈亚西撰写，今日起已在各大书店上架！

比尔·菲利普斯——那个至今仍在经济讨论中占据核心地位的"菲利普斯曲线"的创造者——在研究经济时采用了一种极具创造性的方法。菲利普斯出生于新西兰，二战结束后在伦敦学习经济学，此前他曾捕猎鳄鱼、开采金矿，还当过电气工程师。

他的创造力有一个例子可以证明：1949年，菲利普斯真的在自己的车库里建造了一台类似鲁布·戈德堡的机器，水在不同的水箱和腔室之间来回流动。他说这是英国经济的模型。

他在伦敦政治经济学院展示了这个模型，据他的朋友、经济学家理查德·利普西说，"所有教职员工都出来羞辱这个自以为是的傻瓜"。菲利普斯展示了水如何从财政部水箱流向代表医疗和教育的腔室（当政府花钱时），然后当你拉动杠杆调整税率时，水或多或少会被迅速泵回财政部。"十分钟后，事实证明他比在场的所有人都懂得多，他们闭嘴了，"利普西说。他们为他在伦敦政治经济学院提供了一份工作。

菲利普斯很快发现自己与其他经济学家一起喝雪利酒，并辩论新兴宏观经济学领域的原理——宏观经济学不仅研究个人层面的增长和生产力，还研究整个国家层面的增长和生产力。他痴迷于弄清楚如何让市场经济中的生活不那么混乱。

经济史是一个繁荣与萧条的故事。泡沫与破裂。二战和大萧条之前，股市崩盘和各种"恐慌"（1873年恐慌、1907年恐慌）大约每20年发生一次，然后更频繁。总体而言，在增长期甚至快速增长期之间，夹杂着大量的恐慌。

经济学家称这种增长与收缩的摇摆模式为商业周期。大萧条之后，约翰·梅纳德·凯恩斯提出，政府可以防止或至少缓和这些繁荣与萧条。凯恩斯的个人生活充满了与文学界领袖人物的风流韵事。他的主要见解之一是，政府本身可以成为一种平衡力量，像配重一样增加或减少支出。

经济是否非理性繁荣，有大量快速、昂贵的招聘和新投资？那么凯恩斯认为政府应该像父母在喧闹的派对上调低音乐一样减少支出。经济是否在挣扎，有裁员和谨慎的投资者不敢放贷？那么凯恩斯认为政府应该增加支出，注入一些新的活力来提振招聘和商业活动。

但菲利普斯和他的同行们感觉到通货膨胀也发挥了作用，因为它与失业有关。当失业率低时，工人有更多的议价能力要求加薪。如果拉丁语导师和小部件工厂工人得到更多报酬，就会推高语言课程和小部件的价格，这样就会导致通货膨胀。这种通货膨胀理论后来被称为工资-价格螺旋。（另一种抑制通货膨胀和经济过热的方法是增税，这样公司和家庭就会减少招聘和支出。奇怪的是，这个想法从未流行起来！）

一天，一位教授同事告诉菲利普斯，他拥有测试该理论的完美数据集：英国100年的工资和失业数据。菲利普斯把数据带回家并进行了研究。

周一，他带着一张图表回来了，利普西描述说这张图表让全场惊叹。他绘制了数据并创建了一条优雅的曲线，显示了失业率与工资增长率之间的反比关系，而工资增长率是通货膨胀的主要贡献者。这似乎证实了就业与通货膨胀之间存在明确的权衡，通货膨胀率的变化会改变就业率，反之亦然。高通货膨胀和低失业率是并存的。

菲利普斯的图表传播开来。其他经济学家在其他国家的数据中也发现了同样的曲线，包括美国。1961年，保罗·萨缪尔森和罗伯特·索洛将其命名为菲利普斯曲线，萨缪尔森将其收入他的教科书《经济学》中。在白宫，顾问们引用菲利普斯曲线来描述就业与通货膨胀之间的这种权衡。政府开始几乎将其视为手册——政策制定者可以在曲线上选择一个点并以此为目标。

它的主要使用者是中央银行家。在英国，中央银行是英格兰银行。在美国，是美联储。一般来说，管理通货膨胀是中央银行的责任。然而，当菲利普斯首次绘制他的图表时，即使是中央银行家对通货膨胀也只有初步的了解，因为货币本身正处于一个新时代。直到20世纪30年代和大萧条，许多国家还使用金本位制——允许任何人将货币兑换成固定数量的黄金——作为"支持"货币并确保其至少保持一定价值的一种方式。今天，我们拥有法定货币：这种货币除了相信政府会负责任地管理货币供应并且不会搞砸得太厉害之外，没有任何支持。菲利普斯曲线成为管理通货膨胀的手册，至少直到20世纪70年代，当时美国经济开始表现……不同。经济学家被迫调整和更新反通货膨胀的策略。但那是另一个故事了。

MCP Eats 72% of Your Context Window at Startup
A standard MCP setup can consume around 72% of your available context window before the agent takes a single action. Connect three servers — GitHub, Playwright, and an IDE integration — on a 200K-token model, and tool definitions alone occupy roughly 143K tokens. The agent hasn’t done anything yet. It’s already three-quarters full.
The cost isn’t just tokens. The more unrelated content packed into context, the weaker the model’s focus on what actually matters. A hundred tool schemas sitting in context means the agent wades through all of them on every decision. Researchers have documented what they call context rot — degraded reasoning quality from context overload. In measured tests, tool selection accuracy dropped from 43% to below 14% as tool count increased. More tools, paradoxically, means worse tool use.
The root cause is architectural. MCP loads all tool descriptions in full at session start, regardless of whether the current conversation will ever use them. That’s a protocol-level design choice, not a bug — but the cost scales with every tool you add.
Agent skills take a different approach: progressive disclosure. At session start, an agent reads only each Skill’s metadata — name, one-line description, trigger condition. A few dozen tokens total. The full Skill content loads only when the agent determines it’s relevant. Think of it this way: MCP lines up every tool at the door and makes you choose; Skills gives you an index first, full content on demand.
CLI tools offer a similar advantage. An agent runs git --help or docker --help to discover capabilities on demand, without preloading every parameter definition. Context cost is pay-as-you-go, not upfront.
At a small scale, the difference is negligible. At production scale, it’s the difference between an agent that works and one that drowns in its own tool definitions.
MCP’s Passive Architecture Limits Agent Workflows
MCP is a tool-calling protocol: how to discover tools, call them, and receive results. Clean design for simple use cases. But that cleanness is also a constraint.
Flat Tool Space with No Hierarchy
An MCP tool is a flat function signature. No subcommands, no awareness of session lifecycle, no sense of where the agent is in a multi-step workflow. It waits to be called. That’s all it does.
A CLI works differently. git commit, git push, and git log are completely different execution paths sharing a single interface. An agent runs --help, explores the available surface incrementally, and expands only what it needs — without front-loading all the parameter documentation into context.
Skills Encode Workflow Logic — MCP Can’t
An Agent Skill is a Markdown file containing a standard operating procedure: what to do first, what to do next, how to handle failures, and when to surface something to the user. The agent receives not just a tool but an entire workflow. Skills actively shape how an agent behaves during a conversation — what triggers them, what they prepare in advance, and how they recover from errors. MCP tools can only wait.
